<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| git repo → operator environment | Compose file is committed; env vars must NOT carry secrets in committed files |
| `.env` file → container environment | Docker Compose injects `${VAR}` values from operator `.env` at container start |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01-01 | Information Disclosure | `platform/docker-compose.yml` hardcoded `postgres:postgres` | mitigate | Replace all five occurrences with `${POSTGRES_PASSWORD}` / `${DATABASE_URL}` (Task 1). Eliminates credential exposure in VCS. Finding: F-05. |
| T-04-01-02 | Information Disclosure | `platform/.env.example` using `postgres:postgres` as placeholder | mitigate | Update placeholder to `replace_with_strong_db_password` and add production warning (Task 2). Reduces risk of operators running with default credentials. |
| T-04-01-03 | Denial of Service | `${POSTGRES_PASSWORD}` with no default: empty value causes PostgreSQL auth failure | accept | This is the intended behavior — fail visible rather than fail silently. `.env.example` production warning addresses operator awareness. Rationale: silent empty password is worse than startup failure. |
</threat_model>
