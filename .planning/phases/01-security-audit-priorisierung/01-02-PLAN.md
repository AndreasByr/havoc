## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Filesystem Scan → Scratch-Doc | ripgrep liest gesamtes `platform/`, schreibt Text nach `scratch/grep-results.md`. Keine externen Eingaben, keine Ausführung von Code. |
| Scratch-Doc → Downstream-Waves | Wave 4 liest diesen Output. Muss vollständig sein — fehlende Pattern-Ergebnisse führen zu blinden Spots im Audit. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-06 | T (Tampering) | Unvollständiger grep-Output (Pattern übersprungen oder mit Fehler abgebrochen) → fehlende Findings in Audit | mitigate | Jeder rg-Call endet mit `|| true` damit 0-matches nicht den gesamten Scan-Run killen. Explizite Known-Positive-Presence-Checkliste pro Pattern als Sanity-Gate — Wave 4 liest die Checkliste, nicht nur den Raw-Output. |
| T-01-07 | I (Information Disclosure) | Grep-Output enthält Klartext-Secrets (z.B. echtes Passwort statt Placeholder) | mitigate | P-3-Regex enthält Negative-Lookahead `[^$\{\}]` der `${VAR}`-Interpolationen ausfiltert. Trotzdem: scratch/grep-results.md bleibt im privaten Repo und darf nicht gepusht werden (gleiche Regel wie Audit-Doc). |
| T-01-08 | D (Denial of Service) | ripgrep auf zu großem Directory hängt / frisst RAM | accept | ripgrep ist für Millionen-Datei-Codebases ausgelegt; `platform/` ist ~3k TS/Vue Files — kein Risiko. node_modules/.nuxt/.output sind ausgeschlossen. |
| T-01-09 | R (Repudiation) | Scan-Zeitpunkt unklar → Wave-4-Researcher kann nicht feststellen, ob Codebase inzwischen drifted | mitigate | `**Commit-SHA beim Scan:**`-Zeile im Header + `**Scan-Datum:**`. |
</threat_model>
